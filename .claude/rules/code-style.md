# 代码风格规则

## 命名

- **类型/接口/枚举**：`PascalCase`
- **函数/方法/变量**：`camelCase`（Rust 用 `snake_case`）
- **常量/静态**：`SCREAMING_SNAKE_CASE`
- **文件命名**：组件用 PascalCase（`UserProfile.tsx`），工具用 camelCase（`api.ts`）
- **文件名与默认导出名一致**

## 格式

- 遵循项目已有格式化配置（prettier / rustfmt / eslint）
- 不提交格式化争议，交给自动化工具解决
- 导入分组：外部库 → 本地模块，空行分隔

## 错误处理

- 统一错误类型，避免散落各处的 `unwrap()` / `expect()`
- 错误消息包含足够上下文（操作名、关联 ID、文件名等）
- 不吞没错误，除非有明确理由并注释说明

## 注释

- 只注释"为什么"（why），不注释"是什么"（what）
- 文档注释用于公开 API 签名
- 复杂逻辑或隐含约束必须在代码中注释

## 通用原则

- 不写无关注释，不重构没问题的代码
- 不添加需求之外的功能和抽象
- 删除修改导致的孤儿 import/变量，不删已有的死代码
- 新代码匹配项目现有风格
