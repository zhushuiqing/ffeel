# 测试规范

## 位置

- 测试紧贴被测代码存放（内联或同目录）
- 集成测试放在独立的测试目录
- 测试文件命名：`xxx.test.ts` / `xxx.spec.ts` / `xxx_test.rs`

## 命名

- 模式：`test_预期行为`（`test_can_create_user`, `test_duplicate_key_fails`）
- 描述行为而非实现

## 结构

```
被测函数/模块的测试
├── 正常路径（happy path）
├── 边界条件（空值、极限值）
└── 错误路径（异常、超时、权限不足）
```

每个用例遵循 Arrange → Act → Assert 结构。

## 要求

- 公开函数必须有测试覆盖
- 边界条件和错误路径必须测试
- 不 mock 远程服务——用本地测试替身或真实实例
- 测试之间不共享可变状态，不依赖执行顺序

## 运行

使用项目配置的测试命令运行（`cargo test` / `vitest` / `jest` / `go test` 等）。
提交前确保所有测试通过，失败不允许提交。

## TDD 原则

- 修复 bug 前先写复现该 bug 的测试
- 新功能先定接口再写实现
